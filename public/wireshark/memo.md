# wiresharkメモ

## TCP ZeroWindow

TCP ZeroWindow は、受信側TCPの受信バッファが一杯になり、これ以上受け取れないときに発生します。
要するに「いまは送らないで」という受信側から送信側への制御です。

主な発生パターン

- 受信アプリがデータを読む速度より、到着速度が速い
- 受信側プロセスが一時停止・高負荷・ロック待ちで読み取りできない
- 受信バッファが小さい、または処理キューが詰まっている
- 大量データを短時間に受けるバースト通信
- 受信側装置のCPU/メモリ逼迫やタスク優先度の問題

補足

異常そのものではなく、TCPの正常なフロー制御動作です。
ただし ZeroWindow が長引くと、送信側タイムアウトや切断（再送増加、最終的な通信失敗）につながることがあります。
今回のような現象では、受信側が「いつ・どの処理で読み取り遅延したか」をログで追うのが有効です。

## TCP Retransmission

TCP Retransmission は、送ったTCPセグメントの再送です。
「相手から ACK が来ない（または遅い）ので、同じデータをもう一度送った」ことを示します。

よく起きる理由

- パケットロス（途中で落ちた）
- 相手が忙しく ACK が遅延した
- 回線遅延・ジッタ増大で再送タイマーが先に切れた
- ZeroWindow が続いて送信側が不安定になった（Window Probe/再送が増える）

補足

Wiresharkの TCP Retransmission は解析上のラベルです。
再送そのものはTCPの正常な回復機能です。
ただし頻発するなら、通信品質低下か受信側処理詰まりの兆候です。

受信側が詰まる → ACK遅延/送達停滞 → 再送増加 の流れは十分あり得ます。

### SSLハンドシェイク中のACK2つ

``` text
L.4 TLSv1.2	130	Client Hello
L.5 TCP	60	443 → 1834 [ACK] Seq=1 Ack=77 Win=65535 Len=0
L.6 TLSv1.2	1468	Server Hello
L.7 TCP	1468	443 → 1834 [ACK] Seq=1415 Ack=77 Win=26883 Len=1414 [TCP PDU reassembled in 93011]
L.8 TLSv1.2	1120	Certificate, Server Hello Done
```

L.5とL.7は、ACK番号（Ack=77）自体は同じで、意味も同じです。
どちらも「クライアントからの Client Hello までを受領した」という累積ACKです。

違いはここです。

L.5（93008）
純粋な ACK-only（Len=0）
サーバはまだアプリデータを載せず、受領確認だけ返している
Seq=1 のまま

L.7（93010）
データ付きACK（Len=1414）
サーバの TLS データ（Server Hello 系）を送りつつ、同時に Ack=77 を返している
そのため Seq が 1415 に進んでいる（サーバがデータ送信した分だけ進む）

要するに、
ACK番号の意味は同じ、
L.5は「確認だけ」、L.7は「確認＋データ送信（ピギーバックACK）」です。

## Alert

``` text
L.43 TCP	60	443 → 1834 [FIN, ACK] Seq=3902 Ack=77 Win=26883 Len=0
L.44 TCP	60	1834 → 443 [RST] Seq=77 Win=0 Len=0
L.45 TCP	60	1834 → 443 [RST] Seq=77 Win=0 Len=0
```

L.43の Alert は、TLS の制御メッセージです。

意味は次のとおりです。

- サーバーが TLS セッションを正常に閉じる意思を通知した
- 異常エラー通知ではなく、TLS の終了通知

この直後の FIN, ACK とセットで「TLSとTCPを順にクローズ」する流れ
つまり、この Alert 自体は「暗号化通信を終わります」という合図です。
失敗の根因を示すエラー通知というより、サーバー側が打ち切り判断して終了処理に入ったことを示しています。

## TCP Previous segment not captured

Wireshark の TCP Previous segment not captured は、直前までに見えているシーケンス番号の続きではない TCP セグメントを受信し、途中のデータがキャプチャ上で抜けていると判断したときに出ます。要するに、通信相手がそのセグメントを送ったこと自体は見えているが、その前にあるはずのセグメントをキャプチャできていない、という意味です。

発生しやすいのは次のような場合です。

キャプチャ開始前から通信が進んでいた場合
キャプチャ装置や PC 側で取りこぼしがあった場合
SPAN ポートやミラーポートで一部パケットが落ちた場合
キャプチャフィルタの都合で必要なパケットが入っていない場合
実ネットワーク上で欠落や順序入れ替わりが起き、しかもその欠落区間をキャプチャできていない場合
重要なのは、これは「実通信で必ずパケットロスが起きた」と断定する表示ではなく、「このキャプチャには前のセグメントが見えていない」という Wireshark 側の観測結果だという点です。原因は回線上のロスとは限らず、キャプチャ地点の都合でも起こります。

はい、意味はあります。ただし、これも Wireshark の解析ラベルであって、単独で異常を断定するものではありません。

## TCP Window Full

TCP Window Full は、送信側が相手から広告された受信ウィンドウをほぼ使い切る位置までデータを送った、という意味です。要するに「相手が今受け取れると言っている上限まで、送信データが到達した」という表示です。これはフロー制御上の状態を示しているだけで、正常通信でも普通に出ます。

``` text
L.1 TLSv1.2	130	Client Hello
L.2 TCP	60	443 → 3937 [ACK] Seq=1 Ack=77 Win=65535 Len=0
L.3 TLSv1.2	61	Alert (Level: Warning, Description: Close Notify)
L.4 TCP	60	443 → 3926 [FIN, ACK] Seq=3902 Ack=77 Win=26883 Len=0
L.5 TCP	60	3926 → 443 [RST] Seq=77 Win=0 Len=0
L.6 TCP	60	3926 → 443 [RST] Seq=77 Win=0 Len=0
L.7 TLSv1.2	1468	Server Hello
L.8 TCP	60	3937 → 443 [ACK] Seq=77 Ack=1415 Win=2048 Len=0
L.9 TCP	688	[TCP Window Full] [TCP Previous segment not captured] 443 → 3937 [ACK] Seq=2829 Ack=77 Win=26883 Len=634
```

上記の例では、L.8 でACK=1415 を返しています
受信側の受信ウィンドウは 2048 です
なのでサーバが送れる上限はおおよそ 1415 から 2048 byte 先、つまり Seq=3463 手前までです
実際にその行は Seq=2829, Len=634 なので、2829 から 3462 までを埋めています
つまりこの TCP Window Full は、「サーバが受信側の受信ウィンドウの上端まで送っている」ことを示しており、数値的にも整合しています。

一方で同じ行に TCP Previous segment not captured も付いているので、このケースで重要なのは次の解釈です。

サーバは 1415 の先も送っている
ただし 1415 から 2828 までのどこかのセグメントがキャプチャ上で見えていない
その結果、Wireshark は「前のセグメントが欠けている」と判断した
さらに今見えている 68387 のセグメントは、ちょうど受信ウィンドウの端まで届いているので TCP Window Full も併記された
